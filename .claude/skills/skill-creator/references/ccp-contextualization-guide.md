# CCP Contextualization Guide for Generated Skills

How to contextualize the Code Change Protocol (CCP) and Coding Anchor Points (CAP)
in skills generated by skill-creator.

## CCP in Skill Context

CCP applies whenever the skill **modifies code or configuration**. Skills that only
produce documents (analysis, writing) may have reduced CCP scope, but must still
document why.

### CCP-1 (Intent) — Always Required

Every skill protocol must begin with intent clarification:

```markdown
## Protocol

### Step 1: Intent Clarification (CCP-1)
Before execution, confirm:
- What is the primary output?
- What constraints apply?
- What is out of scope?
```

### CCP-2 (Impact Analysis) — Code-Touching Skills

Skills that modify code must include impact analysis:

```markdown
### Step 2: Impact Analysis (CCP-2)
Before modifying any file:
- Identify direct dependents (callers, importers)
- Check Hub-Spoke sync targets (if editing Hub files)
- Verify test files that need updates
- P1 support: ccp_ripple_scanner.py auto-discovers dependencies at Edit/Write time
```

### CCP-3 (Change Design) — Code-Touching Skills

```markdown
### Step 3: Change Plan (CCP-3)
Propose phased changes:
- Phase 1: Core modification (which file/function first)
- Phase 2: Dependency propagation (callers, importers)
- Phase 3: Test/documentation sync
```

## CAP Anchoring

Map Coding Anchor Points to the skill's domain:

| CAP | Generic Meaning | Skill Contextualization |
|-----|----------------|----------------------|
| CAP-1 (Think before coding) | Plan before implementation | {How this skill plans: e.g., "Analyze document structure before writing"} |
| CAP-2 (Simplicity first) | Minimal complexity | {What simplicity means: e.g., "Use existing patterns, don't create new abstractions"} |
| CAP-3 (Goal-driven) | Focus on user's goal | {What the goal is: e.g., "Produce publication-ready output"} |
| CAP-4 (Surgical changes) | Minimal footprint | {What minimal means: e.g., "Edit only the target section, preserve surrounding context"} |

## When CCP Does NOT Apply

Document the exemption explicitly:

```markdown
## CCP Scope
This skill produces {analysis/writing/translation} output only.
CCP-2 (Impact Analysis) and CCP-3 (Change Design) are not applicable
because this skill does not modify existing code or configuration.
CCP-1 (Intent) is applied in Step 1 of the Protocol.
```

## Proportionality Rule

Match CCP depth to change magnitude:

| Change Scale | CCP Depth |
|-------------|-----------|
| Trivial (typo, comment) | CCP-1 only — confirm "no ripple" |
| Standard (function/logic) | Full CCP-1/2/3 |
| Architectural (API, schema) | Full CCP-1/2/3 + user approval before execution |

## Example: Contextualized CCP for a Data Pipeline Skill

```markdown
## Inherited DNA — CCP Contextualization

| CCP Phase | Application in Data Pipeline |
|-----------|---------------------------|
| CCP-1 Intent | Define pipeline input/output schema, data quality requirements |
| CCP-2 Impact | Check downstream consumers of pipeline output, verify schema compatibility |
| CCP-3 Design | Phase 1: Modify extractor → Phase 2: Update transformer → Phase 3: Verify loader |
| CAP-1 | Design pipeline DAG before writing code |
| CAP-2 | Use standard ETL patterns, avoid custom orchestration |
| CAP-3 | Output must match consumer's expected schema exactly |
| CAP-4 | Modify only the affected pipeline stage, not the entire DAG |
```
